## v0.12.1 — SavedData save main-thread deep-copy spike fix

Fixes a performance issue: when autosave writes SavedData (`.dat` files — scoreboards, raid data, forced-chunk tables, and mod-registered saved data), the main thread deep-copies the entire NBT tree; for large SavedData this step can take seconds, showing up as a sharp TPS drop / stall at every autosave. Servers running mods with large SavedData (e.g. MTR) are hit hardest

### The problem

When v0.12.0 fixed the SavedData async-write race (alongside #8), it deep-copied the mod's returned tag with `CompoundTag.copy()` on the main thread before handing it to the background worker, in order to decouple from the live references the mod holds. That deep copy rebuilds a parallel NBT object tree on the heap (heavy `Object2ObjectOpenHashMap` allocation), with a cost proportional to the SavedData size; for a single very large SavedData this one step alone burns seconds of a tick, and the worker then re-serializes that copy — redundant work

Measured in a profile: on one production server this deep copy took 7.2 seconds in a single pass, 33% of the main-thread time on the stalling ticks

### The fix

The main thread no longer deep-copies the whole tag; it serializes it once into uncompressed NBT bytes. The decoupling is equivalent and in fact stronger — bytes can no longer reference the mod's live subtree — but no parallel object tree is allocated, so the main-thread cost drops sharply; the worker gzips those bytes straight to disk, dropping the "copy + re-serialize" redundancy. The on-disk content is byte-for-byte unchanged

The chunk / entity save paths are unaffected

### Upgrading

Drop in the jar; no config changes. Recommended for servers running mods with large SavedData

Fixes #12
