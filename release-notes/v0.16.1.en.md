# v0.16.1 — Entity terminal relay fix + NeoForge 1.21.1 now published

0.16.1 is a data-safety hotfix: it closes the one entity-save path that, on retry-exhausted terminal IO failure, dropped the pending relay snapshot instead of re-offering it, preventing the latest entity increment from being silently lost during unload when disk writes keep failing. NeoForge 1.21.1 is now published alongside Forge 1.20.1 after dual-loader real-server regression testing

## What's fixed

Entity save has four paths that relay the latest entity list to disk after an in-flight collision. Until now only the retry-exhausted terminal path (FAILED_TERMINAL) took the pending relay snapshot and discarded it, instead of re-offering it like the other three. The trigger: a chunk's entities are edited at the moment of unload, its first-generation write fails repeatedly until the retry budget is exhausted, and a newer generation is already registered as pending — at that point the entities have been evicted from memory by vanilla, and the entity path has no coordinate recovery queue to fall back on, so the latest increment (named mobs, armor stands, item frames, and the like) is permanently and silently lost

This build makes that terminal branch fully symmetric with the chunk side's handleTerminalFailure: when a pending relay snapshot exists it is re-offered once, inheriting the exhausted retry budget so the cascade converges in one step and never retries forever under a persistent IO fault; only when there is nothing to re-offer does it log an ERROR and finish. The fix is byte-for-byte identical across Forge and NeoForge

## NeoForge 1.21.1 now published

The NeoForge 1.21.1 build was previously held back from releases for lack of real-server regression testing. Ahead of this version both loaders were verified end-to-end on a real server (a Forge 1.20.1 test server and a NeoForge 1.21.1 server: load, save, entity counters, and shutdown drain all clean), so from 0.16.1 NeoForge 1.21.1 ships alongside Forge 1.20.1. The NeoForge build does not include async chunk loading (that feature is Forge-only)

## Verification

- Two new regression tests for the terminal relay landing and the one-step cascade convergence; both loaders' test suites pass
- Dual-loader real-server end-to-end: all workers drained cleanly (97 ms / 90 ms), the entity path had zero failures and no gauge leak

## Upgrading

- Drop-in jar replacement, no config changes
- Recommended for servers with high unload churn (large farms, mob grinders, frequent teleports) and dense entities
