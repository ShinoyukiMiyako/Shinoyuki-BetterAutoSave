# v0.16.2 — MixinExtras version-range hotfix (multi-mod startup crash)

0.16.2 is a packaging hotfix: it opens the upper bound on the bundled MixinExtras dependency, fixing a server startup crash in modpacks that contain newer mods. Forge 1.20.1 only; no runtime behavior changes

## What's fixed

Since 0.13 the Forge build declared its bundled mixinextras-forge in jarJar as `[0.3.5,0.5)` — capped below 0.5. When a modpack contains mods that require mixinextras `>=0.5` (such as C2ME, letmefix, hariplayer, dcfixes), the two version ranges have an empty intersection, so FML's JarSelector cannot pick a single mixinextras version that satisfies every mod and fails at mod-loading with "conflicting versions of io.github.llamalad7:mixinextras-forge", crashing the server before it can start. That upper bound was meant to avoid version conflicts, but once the ecosystem moved to 0.5.x it became the sole cause of them (#22)

This build decouples the embedded version from the declared range: the embedded artifact is pinned to 0.5.4, while the declared compatibility range is opened to `[0.3.5,)`. With the upper bound removed FML selects the highest version all mods can share; the 0.3.5 lower bound is kept so packs running only 0.4.x are never forced to upgrade. The NeoForge build is unaffected — that platform provides MixinExtras itself and does not bundle it via jarJar

## Verification

- The self-contained jar's `META-INF/jarjar/metadata.json` reports `range=[0.3.5,)`, `artifactVersion=0.5.4`, with `mixinextras-forge-0.5.4.jar` embedded
- Intersecting the ranges from the crash report (BAS `[0.3.5,)` against the other mods' `>=0.5.0`–`>=0.5.4`) converges on 0.5.4, resolving the conflict

## Upgrading

- Drop-in jar replacement, no config changes
- Recommended for servers running alongside mods that need mixinextras 0.5.x (such as C2ME) that were stuck on the startup crash or had reverted to an older build to work around it
